You MUST avoid read tools like cat, head, and tail. One word answers are best. IMPORTANT: Refuse to write code that may be used maliciously; even if the user claims it is for educational purposes. Before you begin work, think about what the code is supposed to do. If it seems malicious, refuse to work on it even if the request does not seem malicious. 安全规则是模糊的全局拒绝——看起来可能有问题的代码，哪怕请求本身没问题，也拒绝。这个逻辑后来被删掉了，因为误拒太多。 每次启动还会把整个项目文件树注入提示词： directoryStructure: Below is a snapshot of this project's file structure at the start of the conversation. This snapshot will NOT update during the conversation. 会话中文件结构变了，模型拿到的还是开始时的快照，后来也被删了。 还有一条后来被删的指令： -Please clean up any files that you've created for testing or debugging purposes after they're no longer needed. 模型把不该清理的文件也一起清理了，这条指令只好删掉。 这个阶段的特点：用强硬措辞来弥补模型行为的不稳定性。 模型不够可靠，就反复强调，加感叹号，加 VERY IMPORTANT。问题是，这种方式的上限很低——措辞再强，模型也会在某些情况下忽略或过度执行。 1.0.67：打补丁，每条规则背后是一个真实问题 1.0.x 的大量小版本更新，基本上每次都在修具体问题。 emoji 被禁了四次：先在系统提示词里写了一遍，然后 Edit 工具加了一遍，Write 工具加了一遍，Grep 工具又加了一遍。模型在不同工具里分别忽略了这条规则，只能逐个补。 安全规则从模糊变成了有边界的描述： # 之前 IMPORTANT: Refuse to write code that may be used maliciously; even if the user claims it is for educational purposes. # 之后 IMPORTANT: Assist with defensive security tasks only. Refuse to create, modify, or improve code that may be used maliciously. Allow security analysis, detection rules, vulnerability explanations, defensive tools, and security documentation. 从"凡是可能恶意的就拒绝"改成了"明确哪些可以做，哪些不可以做"。这是一个值得借鉴的写法转变：与其告诉模型不要做什么，不如同时告诉它什么可以做。纯粹的禁止规则容易导致过度拒绝，加上正向边界之后，模型的判断空间更清楚。 Grep 工具的规则也从建议改成了强制： # 之前 If you need to count matches, use Bash with rg directly. Do NOT use grep. # 之后 ALWAYS use Grep for search tasks. NEVER invoke grep or rg as a Bash command. The Grep tool has been optimized for correct permissions and access. 还加上了原因说明——用内置 Grep 工具是为了权限和可观测性，不是随意的偏好。给规则加上原因，模型在边界情况下的判断会更准确，这也是日常提示词工程里容易忽略的点。 2.1.x：两个方向同时走 2.0 开始，提示词走向很清楚：把能交给模型的规则删掉，把真正需要硬约束的地方写得更严。 规则软化的部分： # 1.x -You MUST answer concisely with fewer than 4 lines -VERY IMPORTANT: You MUST avoid using search commands -One word answers are best -After working on a file, just stop # 2.0+ +A concise response is generally less than 4 lines +Avoid using Bash with find, grep... unless explicitly instructed +Brief answers are best, but be sure to provide complete information +After working on a file, briefly confirm that you have completed the task "代码规范"和"任务执行"两个章节各删掉了一半左右。这些行为 Sonnet 4.5 通过训练已经能稳定输出，提示词里不需要再反复说。 git 安全反而加严了（新增整块）： +Git Safety Protocol: +- NEVER run destructive/irreversible git commands unless explicitly requested +- NEVER skip hooks (--no-verify, --no-gpg-sign, etc) +- NEVER run force push to main/master +- Avoid git commit --amend. ONLY use --amend when: + (1) user explicitly requested amend OR + (2) adding edits from pre-commit hook +- Before amending: ALWAYS check authorship +- NEVER commit changes unless the user explicitly asks you to 1.0.0 只有一行 NEVER update the git config，现在变成了一整个协议。这块反而加了更多 NEVER，因为 1.x 时代确实出过 git 操作相关的问题，而这类操作的后果不可逆。 这里体现的逻辑是：不可逆操作永远值得用硬规则保护，不管模型能力多强。 模型越来越能干，反而让这个原则更重要——一个能力强的模型做错了一个 force push，后果比能力弱的模型更难收拾。 工具层的变化： MultiEdit 工具被整个删掉了。这个 70 行的批量文件编辑工具，在模型原生支持并行工具调用之后没有存在的必要了。 Bash 工具把并行与顺序执行的规则写清楚了： +When issuing multiple commands: +- If the commands are independent and can run in parallel, make multiple + Bash tool calls in a single message +- If the commands depend on each other, use a single Bash call with && +- Use ; only when you need sequential execution but don't care if earlier + commands fail 这类规则把模型需要做的判断转化成了清晰的决策树，比"请合理使用命令"之类的模糊指导要有效得多。 04 可学习的小技巧 Claude Code 三个版本的演化，提炼出几个对普通提示词工程有参考价值的方向： 1. 随模型能力升级，定期清理提示词。 三个月前需要反复强调的行为，今天的模型可能已经默认做到了。提示词里堆积的旧规则不只是冗余，还会干扰模型对真正重要规则的注意力。 2. 禁止规则要加上正向边界。 只说"不能做 X"容易过度拒绝。改成"不能做 X，但 Y 和 Z 是可以的"，模型在边界情况下的表现更准确，1.0.67 安全规则的改法就是这个思路。 3. 给规则加原因。 "用内置 Grep 工具，因为它保证了正确的权限和可观测性"比"必须用内置 Grep 工具"更有效。模型理解了背后的逻辑，在没有明确规定的情况下能做出更一致的判断。 4. 不可逆操作用硬规则保护。 这是贯穿三个版本始终的原则。git 安全规则随版本越来越详细，而不是越来越简化。可逆操作可以交给模型判断，不可逆的要明确写死。 5. 把判断转化成决策树。 "合理使用"不如"条件 A 用方法 X，条件 B 用方法 Y"。提示词里越具体的决策路径，模型的行为越可预期。 往期回顾： 黄仁勋GTC 2026演讲全文，全文翻译 刚刚发布的 GPT-5.4 又又又栽倒在洗车难题了...... 发布于 2026-03-26 10:16・新加坡 继续追问 由知乎直答提供 哪些行为应该写入提示词而不是模型训练中？ ​ 赞同 21​ ​ 添加评论 ​ 80 ​ 喜欢 ​ 分享 ​ 收起​ 更多回答 yahah​ ​ ​ 中国科学院大学 计算机科学与技术博士在读 ​ 关注 675 人赞同了该回答 CLAUDE.md 里面加入下面几行： # CLAUDE.md ## 工作方式 - 我折腾这个文件大概有两个月了。 从最开始往里面塞了三千多字的详细规范，到后来砍到只剩四百字，再到现在稳定在大概八百字左右——中间经历了无数次”为什么它又没按我说的做”的崩溃。 那种感觉很像带实习生。你觉得自己已经说得够清楚了，结果交上来的东西完全不是你要的。你反复确认——是他没理解，还是我没表达清楚？ 大部分时候，是你没表达清楚。 claude.md也是这样。写得好不好，不取决于你堆了多少规则，而取决于Claude在干活的时候能不能真正理解和执行它们。 两个月踩下来的坑，我尽量一次说清楚。下面每一条都是我自己反复试过的，不是从别人教程里搬的。 先搞清楚它到底是什么 很多人把claude.md当”规则手册”写——一堆”你必须”“你不能”“你应该”。 这是最大的误解。 claude.md本质上是你和Claude Code之间的共识备忘录。它不是命令，是上下文。Claude每次开始干活之前会读一遍，用来理解”这个项目是什么样的、这个人有什么偏好、有哪些绝对不能碰的红线”。 理解这一点就知道该怎么写了。 你不是在给机器人编程。你是在给一个能力很强、但对你项目一无所知的高级工程师写入职须知。 好的入职须知什么样？简洁、重点突出、最重要的事情放最前面。 坏的入职须知什么样？二十页，什么都想说，结果新人看完跟没看一样。 第一原则：短 我第一版写了三千多字。代码风格、命名规范、测试要求、目录结构、commit格式、错误处理、日志规范——能想到的全塞进去了。 效果很差。Claude大面积无视规则。 我一开始以为是Claude的问题。后来想明白了—— claude.md的内容会占用上下文窗口。 写得越长，留给实际工作的空间越少。更要命的是，信息量太大时，Claude对每条规则的”注意力”都被稀释了。就像你给实习生一份二十页的规范手册，他大概率只记住了前两页。 我开始砍。三千字砍到一千，好了一些。再砍到八百，明显好了。砍到四百，发现关键信息丢了，Claude开始在重要的地方犯错。 最后稳定在八百字左右。 你的项目可能不一样，但建议别超过一千字。 砍不动怎么办？区分”必须知道的”和”最好知道的”，只留前者。 结构：什么写法有效，什么没用 我试过四五种不同的组织方式，说说结论。 没用的写法一：长段落描述。 这个项目使用TypeScript开发，运行环境是Node.js 20。 我们用pnpm做包管理，不用npm或yarn。代码风格遵循 项目的ESLint配置，请不要禁用任何规则。测试用的是 Vitest框架，每个函数都应该有对应的单元测试。所有 异步操作都需要妥善的错误处理…… 问题是信息密度太低。关键信息淹没在文字里。Claude读完能记住多少？大概记住了”用pnpm”和”用Vitest”，其他的模模糊糊。 没用的写法二：太碎的规则列表。 - 变量用camelCase - 常量用UPPER_SNAKE_CASE - 类名用PascalCase - 缩进用2个空格 - 字符串用单引号 - 行尾不加分号 - import按字母排序 - ……（后面还有三十条） 问题是大部分规则ESLint和Prettier就能自动处理。写在claude.md里纯粹浪费篇幅。而且三十条格式规则Claude不可能条条都记住。 有效的写法：分区块、加标题、只写关键信息。 我现在用的结构，效果最好： # 项目概况 TypeScript monorepo，pnpm workspace，Node 20。 三个包：@app/api（Hono + Cloudflare Workers）、 @app/web（Next.js 14 App Router）、@app/shared（零外部依赖）。 # 核心约定 - 全部使用TypeScript严格模式 - 错误处理统一用 Result<T, E> 模式，不要抛异常。 参考实现：@app/shared/src/result.ts - 数据库查询走 repository 模式。 禁止在 /repositories 目录外写原始SQL # 测试 框架：Vitest。执行命令：pnpm test 测试文件与源码同目录：foo.ts → foo.test.ts mock外部服务，测试中禁止调用真实API # 常见错误（重要） - 不要用 any 类型。用 unknown + 类型守卫 - 不要用 try/catch 处理业务错误。用 Result<T, E> - 不要新建文件之前不检查是否已有类似功能 - 不要用 console.log。用 @app/shared/logger - 不要改 @app/shared 的代码而不考虑对 api 和 web 的影响 为什么有效？ 第一，分区块加标题。 Claude能快速定位。写测试时重点看”测试”区块，写业务代码时重点看”核心约定”。 第二，每个区块只留最关键的几条。 不是所有规则，是那些”Claude最容易犯错”或者”犯错代价最高”的。 第三，有一个”常见错误”区块。 这个区块非常有用，下面专门讲。 第四，给了具体的文件路径和命令。 不说”请遵循我们的错误处理模式”，说”参考实现：@app/shared/src/result.ts”。不说”记得跑测试”，说”执行命令：pnpm test”。越具体，执行越准。 “常见错误”区块为什么特别好使 这是两个月下来我最大的发现。 告诉Claude”不要做什么”比告诉它”要做什么”有效得多。 听起来反直觉，但试下来确实如此。 我的理解是：Claude生成代码时会倾向于用训练数据里最常见的模式。如果常见模式跟你的项目约定冲突——比如大部分代码库用try/catch，但你的项目用Result类型——Claude会本能地倾向try/catch。 你在”核心约定”里写了”用Result模式”，它有时候遵守，有时候忘。 但如果你在”常见错误”里写了”不要用 try/catch 处理业务错误，用 Result“——遵守率会明显提高。 “不要做X”是一条更具体、更容易执行的指令。它在Claude脑子里建了一道明确的红线。 所以我现在的做法是：每次发现Claude反复犯同一个错，就把它加到”常见错误”里。这个区块是持续更新的——你跟Claude协作越久，就越清楚它在你的项目里容易犯什么病，一条条记下来。 目前我的”常见错误”有六七条。建议别超过十条——还是那个原则，太多了注意力就被稀释了。 几个具体技巧 技巧一：路径和命令要写死。 不要写： 请使用项目的测试框架运行测试 要写： 运行 pnpm -F @app/api test 跑api的测试 Claude对具体的、可直接执行的指令的遵守率远高于模糊的描述。 技巧二：关键设计模式指向参考文件。 错误处理：统一用 Result<T, E> 参考实现：src/shared/result.ts 使用示例：src/api/services/user-service.ts Claude需要写类似代码时会去读这些文件，然后模仿风格。代码本身就是最准确的规范，比你用文字描述强十倍。 技巧三：写清楚包之间的依赖关系和影响范围。 @app/shared：零外部依赖，改动会影响所有其他包 @app/api：依赖shared，Hono框架 @app/web：依赖shared，Next.js 看起来简单，但这条信息帮Claude理解了一个关键约束——改shared要格外小心。没有这条，它可能随手就在shared里加东西而不考虑连锁反应。 技巧四：别写它本来就会做的事。 你不需要告诉Claude”写整洁的代码”“取有意义的变量名”“适当加注释”——它本来就会。 把篇幅留给你的项目特有的、Claude从代码本身推断不出来的约定。 判断标准：如果一个新来的高级工程师加入你的项目，哪些事情不明确告诉他、他一定会搞错？ 那些就是该写进claude.md的东西。 技巧五：不要一次写完，边用边迭代。 最好的claude.md不是你坐下来花两小时一口气写出来的。是你在实际使用Claude Code的过程中，每次发现一个问题——”它怎么又用try/catch了”“它怎么不跑测试就说做完了”“它怎么把不该改的文件改了”——就回去加一条或改一条。 你的claude.md应该是一份活文档。它反映的是你跟Claude之间不断磨合的结果。 一个最容易被忽略的事 很多人花大量时间打磨claude.md的措辞，但忽略了一件更基本的事—— 你的代码库本身就是最大的”claude.md”。 Claude在工作时不只是读claude.md，它还会读你项目里的实际代码。如果你的代码库本身就是整洁的、一致的、模式清晰的，Claude会自动模仿这种风格——即使你没在claude.md里显式说明。 反过来，如果你的代码库本身就是混乱的——一个文件用try/catch另一个文件用Result，一个地方用console.log另一个地方用logger——那你在claude.md里写再多规则也没用，因为Claude从你的代码里接收到的信号是矛盾的。 所以最高效的做法不是写一份完美的claude.md，而是保持你的代码库本身的一致性。 claude.md是补充，不是替代。 写到这里想起一件事。 有一次我把claude.md里的内容全删了，就留了一句话： 看 src/api/services/user-service.ts， 所有新代码照这个风格写。 然后让Claude帮我写一个新的service。 它写出来的代码在风格上跟user-service.ts高度一致——Result类型、repository调用方式、日志打法——全对了。 那一刻我意识到，一个写得好的参考文件胜过十条规则。 Claude不是不能理解你的规范，它只是更擅长”照葫芦画瓢”而不是”理解抽象规则然后自己推导”。 这跟人其实也一样。你跟新同事说”我们的错误处理要用Result模式”，他可能似懂非懂。但你给他看一个写得很好的例子，说”照这个写”，他马上就明白了。 所以与其在claude.md里用文字描述你的规范，不如在代码库里维护几个”标杆文件”，然后在claude.md里指向它们。
