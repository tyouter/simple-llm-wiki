我描述了我的需求，将其分解成多个并行任务，然后看着四个人工智能程序同时构建它，而我则在一旁审查、调整和完善。最终，功能上线了。它运行正常。说实话，感觉真好。我知道最终结果应该是什么样，我把它描述得很清楚，结果也完全符合预期。当一切顺利完成时，那种感觉真的很令人满足。 但满足感之下，还隐藏着一些难以言喻的东西。与其说是不适，不如说是你与自己创造之物之间关系的悄然转变。你构思了它，指导了它，验证了它。但你并没有亲手创造它。至少不是以你以往的方式。而这改变了你对“属于我”这个概念的感受。 我曾经是一名机器学习工程师，现在是一名技术创始人。我曾花费数千小时编写代码、调试代码、思考代码。而现在，我越来越少地编写代码，而是编写需求、协调流程、评估输出结果。我思考的是需要什么，而不是如何实现它。 本文并非探讨人工智能是否会取代工程师。这类争论枯燥乏味，答案显然是否定的。我想探讨的是一个更令人不安的问题：当我们赖以建立身份认同的技能不再是挑战时，我们会发生什么？我们会失去什么？又会得到什么？当编写代码不再是工作内容时，未来的工程师究竟会是什么样子？ 抄写员谬误 大家都在问错问题。“人工智能会取代工程师吗？”这个问题，就好比是2025版的“印刷机会取代抄写员吗？”答案显然是肯定的。但这并非重点。重点在于接下来发生的事情。 在古腾堡之前，如果你想要一本书，就得雇佣抄写员。这是一项技术性很强的职业。抄写员需要多年的训练，一丝不苟的细节把控，以及对语言和文本的深刻理解。他们的价值是实实在在的，也是靠实力赢得的。但这份工作绝不仅仅是“用羽毛笔在羊皮纸上书写”。它的真正意义在于使知识得以传播。羽毛笔只是当时唯一可用的工具而已。 印刷机出现后，抄写员并没有一夜之间全部消失。但那些幸存下来的人明白，他们真正的技能并非书法，而是懂得哪些内容值得抄写。编辑、翻译、出版商，职能发生了演变，但目的却始终如一。 同样的模式随处可见。电话接线员的工作并非只是把线缆插到电路板上，而是连接人与人。会计的工作并非只是计算数字，而是让财务决策清晰易懂。在每一种情况下，人工环节都被自动化了，而那些最终取得成功的人，正是那些早已领悟到人工环节并非关键所在的人。 现在，对于工程师来说，这才是真正让人感到不舒服的地方。与其他职业相比，我们工程师的身份认同几乎完全建立在技术本身之上。不仅仅是“我用软件解决问题”，而是“我编写优雅的代码”。清晰的架构、完美优化的函数、凌晨两点调试晦涩难懂的代码直到最终成功，这些都能带来真正的满足感。对大多数工程师来说，这并非为了炫耀，而是为了对系统有足够深入的理解，从而能够按照自己的意愿塑造它，这种感觉令人心生自豪。这正是我们最初被这个领域吸引的原因：亲手创造，即使只是敲击键盘，也能带来成就感。 正因如此，这种转变才如此切身相关。大多数工程师都能保住工作。但工作中那些最能带给我们成就感的部分，却最先被自动化取代。剩下的，是知道该构建什么、为什么要构建，以及构建出来的东西是否正确。这是一种截然不同的技能。可以说，它更难掌握。但感觉却不一样。对很多人来说，这远远不够。 我认为这才是目前行业焦虑的真正根源。不是对失业的恐惧，而是对被时代淘汰的恐惧。是害怕自己花费数年时间精通的技能可能不再重要。如果你诚实地面对自己，你会发现这种恐惧并非完全没有道理，只是找错了对象。这门技艺并没有消亡，而是在转型，从编写代码的方式到思考问题的方式都在发生变化。 所有权问题 说实话，我觉得很少有人谈到这一点。当你亲手编写每一行代码时，你对系统的理解几乎就像是切身感受。你知道为什么某个函数要这样构建，因为你事先尝试了三种不同的方法。你知道那些边界情况隐藏在哪里，因为你亲身经历了无数次的尝试才发现它们。代码库不仅仅是你工作的对象，它就像你熟悉自己的公寓一样，即使在黑暗中也了如指掌。 当人工智能编写代码时，那种亲密感就消失了。你可以阅读输出结果，审查它，测试它。但你并没有亲身经历塑造它的那些决策过程。这其中确实存在着一种损失。我不会否认这一点。最初几次，当我只描述我的需求，而让 Claude Code 构建整个模块时，我发现自己会想：“这还是我写的吗？” 我认为答案是肯定的。但我花了一段时间才明白这一点。 对我帮助最大的类比是思考建筑师。建筑师不砌砖，不浇筑混凝土，也不布线。但没有人会走进一栋建筑说“这根本不是建筑师的作品”。建筑师构思了它，设定了限制条件，决定了哪些东西放在哪里以及为什么。价值始终在于远见和判断，而非具体的建造。 电影导演、音乐制作人、产品设计师的情况也是如此。我们通常认为的某项作品的创意所有者，几乎从来都不是那些事无巨细亲力亲为的人。他们只是清晰地把握了创作意图，从而使执行得以顺利进行。 这就是工程师们目前正在经历的转变。所有权正在从“我写了这个”转移到“我定义了这个，验证了这个，并对此负责”。这是一种不同的所有权。它不再那么注重实际操作，而是更注重理性思考。说实话，这需要一段时间来适应。曾经有一种工程师的自豪感，来自于亲身与一个bug搏斗了六个小时。这种自豪感正在消退。取而代之的是一种自豪感，这种自豪感来自于我足够清醒地思考，从一开始就没有意识到这个bug的存在。 我不会假装过渡过程毫不费力。但我也不会假装我怀念花六个小时调试，只为了找到一个三十秒就能改正的拼写错误。新版本更有价值。你只需要先放下旧版本。 当想法变得毫无价值时 这种转变还有第二个后果，我认为它比所有权问题更少被提及，而且更加残酷。 如果软件开发变得极其廉价，那么开发能力就不再是竞争优势。这意味着，创意本身将变得毫无价值。 这听起来可能有点反直觉。多年来，瓶颈一直在于执行。“创意不值钱，执行才是关键”几乎是创业圈的口头禅。如果你有一个应用创意，问题永远是：你真的能把它做出来吗？你具备技术能力、团队、时间和资金吗？大多数创意都止步于此，在执行这道坎上夭折。而正是这道坎赋予了工程师们优势。我们是能够跨越这道坎的人。 如今，这道壁垒正在瓦解。过去需要一个月才能完成的业余项目，现在只需一个周末就能搞定。过去需要全栈开发人员才能完成的原型，现在连电脑都没打开过的人也能轻松搭建。当人人都能动手实践时，动手能力就不再是筛选条件。如果执行力不再稀缺，那么创意自然也不会稀缺。因为每个人原本都有创意，只是以前无法将其付诸实践而已。 那么现在真正稀缺的是什么呢？ 判断。品味。分配。 知道在一百种可能的产品中，哪一种最值得开发；知道你开发的产品是否真正好用，而不仅仅是功能齐全；以及如何将它推向目标用户。这些技能一直都很宝贵，但过去它们属于不同的部门。产品经理思考开发什么产品；设计师思考产品是否好用；市场营销人员思考分销渠道；而工程师则只负责开发产品。 这种界限正在消融。未来的工程师需要跨越所有这些层面进行操作，因为纯粹的构建层不再是价值的集中点。编写代码曾经是护城河。如今，护城河是围绕代码的一切：问题选择、质量判断、市场推广的直觉。 我知道，如果你毕生致力于追求技术深度，这听起来可能会让你感到威胁。但我其实认为这是一种解放。一个人所能取得的成就上限被大幅提升了。你不再受限于打字速度，而是受限于思维清晰度。而且，与打字不同，思维能力会随着经验的积累而提升，而不是随着年龄的增长而衰退。 我现在实际做的事情 让我描述一下我现在典型的工作日是什么样的，因为我认为这比任何抽象的论证都能更好地说明这种转变。 我不再打开 VS Code 了，已经好几周没用了。取而代之的是，我打开了一个名为Clave的工具，这是我专门为这种新的工作方式开发的桌面应用程序。它是一个 Claude Code 的会话管理器，让我可以在同一个窗口中并行运行多个 AI 编码会话。它内置了一个看板，我可以在上面创建任务，然后直接从看板启动每个任务，进入它自己的 Claude Code 会话。 所以我的早晨从思考开始。今天需要构建什么？有哪些依赖关系？哪些任务可以并行运行而不会冲突？我把工作分解成看板上的各个独立任务，为每个任务编写清晰的需求，然后同时启动三到四个会话。之后，我会观察、审查并调整。一个会话是构建一个新的 API 端点。另一个会话是重构一个组件。第三个会话是为我昨天发布的功能编写测试。我的工作是确保每个会话都有足够的上下文信息来成功运行，并及时发现 AI 做出的我意想不到的假设。 在这个工作流程中，最重要的技能不是编码，而是分解。你能否将一个复杂的问题分解成足够独立、可以并行运行，但又足够连贯、最终能够完美集成的模块？这才是系统思维，而不是编程。而且，这真的很难。在很多情况下，它比自己编写代码更难，因为你必须在脑海中把握整个架构，同时还要将细节委托出去。 最令我感到欣慰的是，Clave 本身就是这样构建的。我用 Claude Code 构建了一个管理 Claude Code 的工具。我当然明白其中的递归原理。这个工具正是源于它所旨在实现的流程转变。它的诞生源于我个人的挫败感：我经常需要在终端标签页之间切换，搞不清哪个会话正在做什么，还要在窗口之间复制粘贴上下文。于是，我描述了自己的需求，并像现在构建所有东西一样，用同样的方式构建了它：先提出需求，然后由 AI 执行，最后由人工判断。 我之所以开源，是因为我认为这种工作流程并非我独有。任何经历这种转型的工程师最终都会遇到同样的难题：你需要一种方法来管理你工作中已经演变成的编排层。Clave 就是我的解决方案，但更重要的是，它具体展现了这种转变之后工程工作的面貌：减少打字，增加思考；减少执行，增加指导。 GitHub - codika-io/clave 在 GitHub 上创建帐户，即可参与 codika-io/clave 的开发。 github.com 如果你想具体了解这种转变有多么巨大：我只用了几个小时就开发出了一个桌面应用程序，取代了我多年来每天都在使用的工具 VS Code。这不是一个玩具原型，而是一个真正功能齐全的应用程序，具备多会话管理、看板、主题和快捷键等功能。这在不久前需要花费数周时间才能完成。一个人只需几个晚上就能开发出一个完善的开发者工具，并将其作为主要工作流程，这比任何理论文章都更能说明工程技术的未来发展方向。 明日工程师 如果编写代码不再是工作，那么什么才是呢？ 我认为有必要回顾一下工程的真正含义。字典对工程师的定义是“设计、建造或维护机器、结构或系统的人”。设计。建造。维护。请注意，这里并没有提到代码。工程从来就不是编写代码。它关乎批判性思维、科学方法以及高效地解决问题。代码只是实现层面，是我们用来表达解决方案的工具。这部分工作从未改变。改变的是，一旦问题被理解，解决方案清晰明确，可能就不再需要自己去实现了。 但正是在这里，我对未来感到不确定，我认为理性的诚实比自信的预测更重要。 如今，人工智能毫不留情地揭露你思维中的模糊之处。如果你给出含糊不清的指令，就会得到含糊不清的结果。你必须精确定义你想要的东西，精准地引导方向，并及时纠正每一个错误的假设。思维依然需要敏锐。人类仍然需要承担定义问题和设计解决方案的艰巨认知工作。 但我已经注意到，过去几个月里情况发生了一些变化。人工智能的起点正在提高。当你给人工智能一个定义模糊的任务时，它的默认方向比以前更接近正确。即使你没有详细说明所有信息，模型也能更好地推断意图、填补空白并选择合理的解决方案。 所以我一直在问自己：这种趋势何时才能结束？到那时，我们是否还需要精确定义解决方案？模型是否能够分析问题、评估各种可能的方法，并在无需我们进行架构设计的情况下，根据具体情况选择最有效的方案？我不知道。而且，我对任何声称自己知道答案的人都持怀疑态度。 我对未来的发展方向比较有把握。未来对通才型人才的需求会更大。工程师需要更多地进行产品思考，更多地与客户沟通，更深入地理解他们所解决问题背后的业务背景。价值的重心会转移到更上游的环节：不再是“我们如何构建这个”，而是“我们应该构建什么，以及为谁构建”。至于未来的工程师会更像产品经理、技术战略家，还是某种我们目前还无法命名的全新角色，我真的不知道。 但我宁愿坦诚面对这种不确定性，也不愿假装自己有明确的答案。转变是真实存在的，但方向尚不明朗。我觉得这样也挺好。 工艺不断演进 这篇文章开头，我描述了一个两年前根本无法想象的时刻：一个原本需要几天才能完成的功能，我们只用了一个小时就交付了。这并非靠偷工减料，而是靠一种截然不同的工作方式。 这种转变带来的焦虑是真实存在的，我认为忽视它对任何人都没有好处。如果你花了多年时间精通编程，眼睁睁看着这项技能被商品化，难免会感到不安。这关乎个人，它触及的远不止职业前景，更关乎你如何看待自己。 但我总是会想起抄写员。古腾堡之后那些蓬勃发展的抄写员，并非是那些对鹅毛笔怀念最久的人，而是那些比其他人更早意识到自身真正价值不在于笔迹，而在于懂得什么值得记录在纸上的人。 工程学现在也正在经历同样的认知转变。这门技艺并没有消亡，它只是褪去了我们曾经误认为是核心的外衣。本质上，它的工作依然是：深入理解问题，清晰思考，并确保制造出正确的产品。这始终是工程学的工作。只不过过去我们是用键盘敲击来完成的，而现在我们是用方向盘来完成的。 我不会妄称自己知道五年后会是什么样子。我之前也表达过我的这种不确定性，而且我是认真的。但我知道，我最敬佩的工程师始终是那些思考最深入的人，而不是那些打字最快的人。如果说有什么变化的话，那就是这次的转变正式确立了这种区别。 工程的未来属于思路清晰的人。印刷术从未使优秀的写作过时，它只是让写作变得更加重要。 发布于 2026-02-26 14:59・江苏 ​ 赞同 25​ ​ 1 条评论 ​ 30 ​ 的维护者评价很妙：「如果你曾经想过读博——不如先读完这些文档。」 Parry — prompt 注入扫描器 用 Claude Code 写代码的人越来越多，安全问题随之而来。Parry 是一个 Hook 插件，自动扫描工具输入和输出，检测 prompt 注入攻击、密钥泄露和数据外泄。 还在早期阶段，但方向很重要。AI 工具链的安全，迟早要被认真对待。 Dippy — 解决「权限疲劳」 用 Claude Code 干活，最烦的事情之一就是不停弹权限确认。Dippy 用 AST 解析来自动判断命令是否安全——安全的自动放行，危险的才弹窗。而且同时支持 Claude Code、Gemini CLI 和 Cursor。 Claude ESP — 偷看 Claude 在想什么 一个 Go 写的 TUI 工具，能实时流式展示 Claude Code 的隐藏输出——思考过程、工具调用、子 Agent 状态。开另一个终端窗口就能看。 用来调试特别好使。不用打断主会话，就能知道 Claude 到底在做什么。 AgentSys — 生产级工作流自动化 从任务到生产的全链路自动化——PR 管理、代码清理、性能排查、漂移检测、多 Agent 代码审查。底层用正则和 AST 做确定性检测，LLM 只在需要判断的地方介入。 这个思路很对：能用规则搞定的就别让 AI 猜。 生态爆发背后的三个信号 这些项目多只是表面现象。更值得关注的是背后的趋势。 信号一：Claude Code 正在变成平台 Claude Code 最开始就是个终端工具。但现在围绕它的生态已经长出了： Skills 系统：类似插件市场，SKILL.md 格式已经跨工具通用（Gemini CLI、Codex CLI 也兼容） Hooks 系统：生命周期事件钩子，能写 shell 脚本做确定性控制 Slash-Commands：自定义命令扩展 子 Agent 架构：可以启动独立 Agent 并行处理任务 一个工具有了插件、有了钩子、有了命令系统、有了多进程架构——这就是平台的雏形。 信号二：社区在造「基础设施」 早期生态里的项目大多是 Demo 级的——「看我用 Claude 做了什么有趣的东西」。 现在不一样了。Parry 做安全扫描，Dippy 做权限管理，AgentSys 做生产级自动化。这些不再是玩具，是基础设施。 awesome-claude-code 的目录结构就说明问题。Orchestrators（编排器）、Config Managers（配置管理）、Usage Monitors（用量监控）。这些词放到三年前，是 Kubernetes 生态的词汇。 信号三：AI 编程工具的竞争维度变了 以前 Cursor、Copilot、Claude Code 之间拼的是「谁写代码更准」。现在拼的是生态。 Claude Code 预计到年底将贡献 GitHub 每日 commits 的 20% 以上。官方的 Skills 仓库 8 万多 star。前端设计 Skill 单独就有 27 万次安装。 围绕 Claude Code 的第三方生态越丰富，开发者的迁移成本就越高。想想当年 VS Code 是怎么靠插件生态把 Sublime Text 和 Atom 挤出市场的。 对普通开发者意味着什么 如果你已经在用 Claude Code，有三件事值得现在就做。 第一，去逛一圈 awesome-claude-code。 不用全看，找到和你工作相关的两三个项目就够了。一个好的 Hook 或 Slash-Command 能帮你省掉每天 15 分钟的重复操作。 第二，写自己的 Skills。 Skills 的本质就是 Markdown 文件。把你团队的编码规范、代码审查清单、部署流程写成 Skill，Claude 就能按你的标准干活。一次投入，长期回报。 第三，关注安全。 AI 生成代码越来越多，但代码审查能力没有同步提升。装一个 Parry 之类的安全扫描工具，至少别让明显的漏洞溜过去。 过去一年，AI 编程工具从「能用」变成了「日常」。而 Claude Code 的生态爆发，标志着下一个阶段的开始——从「日常」到「定制化」。 每个团队、每个开发者都在把自己的经验、流程、规范塞进 Claude 里。AI 工具不再是千人一面的通用产品，而是可以按你的方式工作的私人助手。 这才是 AI 编程工具的终极形态吧——不是替代你，是变成你。 参考来源： awesome-claude-code 仓库（31.9k stars）：https://github.com/hesreallyhim/awesome-claude-code
