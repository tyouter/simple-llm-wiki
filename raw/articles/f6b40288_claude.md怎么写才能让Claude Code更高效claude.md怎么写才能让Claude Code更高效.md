这是两个月下来我最大的发现。 告诉Claude”不要做什么”比告诉它”要做什么”有效得多。 听起来反直觉，但试下来确实如此。 我的理解是：Claude生成代码时会倾向于用训练数据里最常见的模式。如果常见模式跟你的项目约定冲突——比如大部分代码库用try/catch，但你的项目用Result类型——Claude会本能地倾向try/catch。 你在”核心约定”里写了”用Result模式”，它有时候遵守，有时候忘。 但如果你在”常见错误”里写了”不要用 try/catch 处理业务错误，用 Result“——遵守率会明显提高。 “不要做X”是一条更具体、更容易执行的指令。它在Claude脑子里建了一道明确的红线。 所以我现在的做法是：每次发现Claude反复犯同一个错，就把它加到”常见错误”里。这个区块是持续更新的——你跟Claude协作越久，就越清楚它在你的项目里容易犯什么病，一条条记下来。 目前我的”常见错误”有六七条。建议别超过十条——还是那个原则，太多了注意力就被稀释了。 几个具体技巧 技巧一：路径和命令要写死。 不要写： 请使用项目的测试框架运行测试 要写： 运行 pnpm -F @app/api Code的过程中，每次发现一个问题——”它怎么又用try/catch了”“它怎么不跑测试就说做完了”“它怎么把不该改的文件改了”——就回去加一条或改一条。 你的claude.md应该是一份活文档。它反映的是你跟Claude之间不断磨合的结果。 一个最容易被忽略的事 很多人花大量时间打磨claude.md的措辞，但忽略了一件更基本的事—— 你的代码库本身就是最大的”claude.md”。 Claude在工作时不只是读claude.md，它还会读你项目里的实际代码。如果你的代码库本身就是整洁的、一致的、模式清晰的，Claude会自动模仿这种风格——即使你没在claude.md里显式说明。 反过来，如果你的代码库本身就是混乱的——一个文件用try/catch另一个文件用Result，一个地方用console.log另一个地方用logger——那你在claude.md里写再多规则也没用，因为Claude从你的代码里接收到的信号是矛盾的。 所以最高效的做法不是写一份完美的claude.md，而是保持你的代码库本身的一致性。 claude.md是补充，不是替代。 写到这里想起一件事。 有一次我把claude.md里的内容全删了，就留了一句话： 看 src/api/services/user-service.ts， 所有新代码照这个风格写。 然后让Claude帮我写一个新的service。 它写出来的代码在风格上跟user-service.ts高度一致——Result类型、repository调用方式、日志打法——全对了。 那一刻我意识到，一个写得好的参考文件胜过十条规则。 Claude不是不能理解你的规范，它只是更擅长”照葫芦画瓢”而不是”理解抽象规则然后自己推导”。 这跟人其实也一样。你跟新同事说”我们的错误处理要用Result模式”，他可能似懂非懂。但你给他看一个写得很好的例子，说”照这个写”，他马上就明白了。 所以与其在claude.md里用文字描述你的规范，不如在代码库里维护几个”标杆文件”，然后在claude.md里指向它们。
