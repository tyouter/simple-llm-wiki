部署坑，这类带浏览器的，其相关库要拉全，要不各种报错。第一步就是花点时间把 docker 镜像弄好，剩下的事儿才好继续。 安全坑，这类浏览器类抓取，一次抓取不仅仅是一个 http 等类型的请求，目标页往往包含十几甚至上百个请求，并且是 chromium net 等浏览器相关 lib 去发的，其基准环境是服务器。因此，时间被请求地址并不是那么可控。为了避免 SSRF 等风险，要做好请求发送前后相关 hook 甚至浏览器内核级别改造（如果有能力做到的话），避免访问页面内不安全地址。 Page loaded 坑，页面完成加载时机实际上没想象的那么准确，特别是在前端渲染为普遍手段的当下。往往 onload 时候页面信息是相对不完整的。导致抓取结果和预期结果差距很大。 截图坑，除了以上页面加载完成点判定外，全页截图可能会存在 WebGL 最大纹理尺寸限制 （1<<14）超了可能崩渲染器，超长页面（实际很普遍）往往无法一次性完整截图。此外，全页截图时可以拉伸视口截图，但拉伸高度根据实例具体配置可能远小于 MAX_TEXTURE_SIZE 值，同样超了可能崩。当然也可以单屏截图依次滚屏后拼接，但如果页面中间有动画、动图、懒加载中间插入等情况，拼出来的就惨不忍睹了。这还没完，现在“到底无限加载”的页面也不少，不一定滚大到底。并且，怎么滚页面也有讲究，单独设置 scroll 去滚，可能无效，因为实际滚动条可能是某个 div 容器的，而不是视口的。当然也可以选择 CDP 里的合成事件去拖拽页面进行滚动，那可能会慢一些，如果能忍受耗时的话。暂时不说这里如果有带内滚的 div 或者 iframe 又要怎么办。 URL 终点坑，一个 URL 并不是一跳就到达最终页面了，它可能 onloaded 之后又跳了一个到N个地址，其余都是中间页，这些中间页可能会加载后直接跳走，也可能会等个 N 秒倒计时（提示给用户）再跳走。如何判定页面是否到达最终地址了呢？ 懒加载坑，在确保页面加载完成判定靠谱，以及跳到最终地址后靠谱后，页面中的懒加载怎么完整的触发？别只认为页面只有下滚的懒加载模式，swiper 这种横向滚动的可能也是懒的。那又要咋定位它咋触发呢？ 页内非图文内容获取坑，页面内有 iframe （还可能多层嵌套）啦，有 video audio 啦，怎么一次行获取全部内容。如果 video 等媒体地址不是一个具体可访问的 URL 呢？比如是个 Blob，又要怎么处理？ 页面抓取快照还原精度坑，快照要显示要欢迎为和原始页面一致，要人看不乱，要链接可被导航，又该如何处理？ 对抗坑，这个涉及内容太多了，UA 和 ip 地址识别性质对抗太常见了。不常见的可能还有非 UA ，的客户端特征识别，比如识别页面视口高度和屏幕高度差异/使用的 CPU 是 ARM 等移动内核/至少有 GPU/不是 Linux 等服务器系统等等（老多一堆前端特征，就不细说了），甚至还针对特定 webview 做 特定 jsb 的识别探针。更坑人的可能还有 ip 代理识别，由于 ip 地理地址的识别并不准确，对同一个代理地址，对抗放的识别地址可能和使用的代理预期地址不同，导致反反爬对抗失败。 友军挖的坑，这也挺多的，稍微说俩。被抓取 URL 和实际预期抓取的 URL 不一致，比如通过内部接口获取的到 URL 只是一个 Base 地址，实际访问时可能会被自己人追加一些 search 参数，导致抓取内容完全不符合预期。还有就是 isp 坑人，比如 isp 会搞一些流量黑洞，或者机房运维给搞一些奇怪的限制，导致某些地址 URL（也可能是代理 ip）完全无结果，可能超时，也可能返回一些莫名其妙的内容。 抓取提速坑，这类浏览器抓取，总要等到页面完事，这个时间说短可真不短了，如果挂了一些不太迅速的代理，搞完一次抓取动作，20s-60s，乃至超过 100s 都有可能。这导致必须要根据具体需求做取舍，做不同承载力的服务。特定场景下，优化好了，2s-10s 内是可以做完一次抓取动作的。但是服务做多了，可能多个服务雷同都很高，代码怎么拆分组织，以及后服务续测试维护么也是个不大不小的坑了。
