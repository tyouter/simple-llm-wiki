个回答 BugBuster喵 ​ 关注 1822 人赞同了该回答 这是我面试高级工程师时很聊的话题之一。 听大家说了很多，我也来聊几个真正让我感到惊艳，甚至改变了我对计算机科学认知的算法。 1. 扔硬币的艺术：HyperLogLog 很多年前，我还在做广告系统的时候，遇到了一个非常经典的需求：统计UV。 如果网站只有几千人访问，这太简单了。你弄个哈希表，来一个IP丢进去一个，最后看集合的大小就行。 但是，当你的用户量是亿级别的呢？ 假设你有1亿个用户，每个用户ID是长整型占8字节。光是存这1亿个ID，就需要几百兆内存。如果你还要统计每天、每小时、不同维度的UV，内存瞬间就会爆炸。 当时我还在死磕怎么压缩数据，后来一位前辈丢给我一篇论文，就是关于HyperLogLog的。 看完我整个人都傻了。这个算法告诉我，统计1亿个基数，只需要12KB的内存，误差还能控制在1%以内。 它是怎么做到的？它的核心思想居然是扔硬币。 想象一下，你和朋友玩扔硬币游戏。如果你说你连续扔出了3次正面，我不觉得稀奇。但如果你说你连续扔出了20次正面，我会断定你肯定扔了成千上万次硬币，或者你就是在作弊。 伯努利试验告诉我们，抛硬币出现连续正面次数越多，说明你需要尝试的实验总次数就越多。 HyperLogLog就是利用了这个概率学原理。它把每个用户ID哈希成一串二进制，这串二进制里的0和1，就相当于硬币的反面和正面。通过观察哈希值中末尾连续零的最大数量，就可以倒推估计出你到底放进了多少个不同的数据。 惊艳点在哪里？ 它完全抛弃了存储数据本身的执念。传统的思维是：我要数人头，我就得把每个人头都记下来。HyperLogLog的思维是：我不记人头，我记人头出现的特征概率。 这种从确定性记录到概率统计的思维跨越，是我职业生涯中第一次感受到数学的暴力美学。它不追求100%的精准，但在海量数据面前，99%的精准度加上极致的资源节省，就是完美的工程解。 如果你对这种概率算法感兴趣，想深入看看它的推导过程（哪怕看不懂公式，看图也好），我强烈推荐你去读一下Redis作者Antirez的博客。他在实现Redis的PFCOUNT命令时写过非常详细的解释。你搜 Redis HyperLogLog explanation 就能找到，那绝对是把复杂理论工程化的典范。 2. 宁可错杀，不可放过：Bloom Filter 接上面的话题。有了UV统计，自然就有去重需求。 比如爬虫系统，我有100亿个URL，来了一个新URL，我怎么知道它是不是已经爬过了？ 或者在垃圾邮件过滤里，怎么判断这个发件人是不是黑名单？ 还是老问题，存不下。 这时候布隆过滤器登场了。它的设计哲学简直就是无赖。 它搞了一个很长的二进制位数组，初始全是0。来一个数据，用好几个不同的哈希函数算出好几个位置，把这几个位置都标记为1。 查询的时候，同样算这几个位置。如果这些位置全都是1，它就告诉你：这东西可能存在。如果有一个位置是0，它就告诉你：这东西绝对不存在。 注意那个可能。 布隆过滤器的惊艳之处在于它承认自己会犯错。它会有误判，可能会把没见过的东西误判为见过的。但是，它绝不会漏判。 在工程实践中，这种有底线的妥协简直是天才的设计。 举个真实的场景：缓存穿透保护。 我们在做高并发系统时，最怕有人疯狂查询一个数据库里根本不存在的key。请求会直接穿透Redis缓存打到MySQL数据库上，瞬间把数据库打挂。 怎么防？我们在Redis前面挡一层布隆过滤器。 所有存在的key都放进布隆过滤器。请求来了，先问布隆过滤器：这key在吗？ 布隆过滤器说：不在。那好，直接拦截，不用查数据库了。 布隆过滤器说：在。那放过去查，虽然有极小概率其实是不在的（误判），但这点漏网之鱼对数据库已经构不成威胁了。 这给了我极大的震撼：完美的算法不一定是没有任何错误的算法，而是能把错误控制在可接受范围内，并换取巨大性能收益的算法。 关于这个，我建议大家不仅要看原理，更要去玩一下可视化演示。Jasper Wang有一个非常棒的Bloom Filter Visualization在线演示网页，你搜一下就能出来。你能亲眼看到哈希冲突是怎么发生的，那种直观的感受比看十页书都管用。 3. 来自高维空间的降维打击：Word2Vec 时间来到2013年左右，那时候深度学习还没像现在这么火。做NLP的人还在搞词袋模型、TF-IDF这些统计方法。 然后Word2Vec横空出世。 当你第一次看到那个著名的公式：King - Man + Woman = Queen 时，你的反应是什么？ 我的反应是：你在扯淡吧？词语怎么能做加减法？ 但当你真正理解了Embedding这个概念时，你会发现这简直是上帝视角。 计算机是不懂中文、英文的，它只懂数字。以前我们给词编码，要么是编号（1是苹果，2是香蕉），要么是One-hot（一长串0，只有一个位置是1）。这些表示方法里，词与词之间是孤立的，没有任何关系。 Word2Vec把每个词映射到了一个高维向量空间里。 在这个空间里，意思相近的词，距离就很近。意思相反的词，距离就很远。词与词之间的关系，变成了向量空间里的方向和距离。 为什么 King - Man + Woman = Queen？ 因为在向量空间里，从King到Man的距离向量，代表了性别剥离。 把这个向量应用到Woman身上，就加上了皇室属性，结果自然就指向了Queen。 这不仅仅是算法的胜利，这是哲学的胜利。它告诉我们，语言的意义不是定义出来的，而是由上下文决定的。 这个算法直接开启了后来的万物皆Embedding时代。现在的推荐系统（比如你刷知乎、刷抖音），底层全是这个逻辑。把你这个人变成一个向量，把视频变成一个向量，算一下余弦相似度，高的就推给你。 想真正理解这个，我强烈推荐Jay Alammar的博客。这位大神是把AI原理可视化的神级人物，他的文章 The Illustrated Word2Vec 是我见过最清晰的讲解，没有之一。我也经常把他的博客发给刚入职看不懂Transformer的小朋友们。 4. 雷神之锤的平方根倒数 这个例子可能有点老，但每次提起来我都得跪着写。 在3D游戏开发中，经常需要计算平方根倒数（1/sqrt(x)），主要用于向量归一化。在早期的硬件条件下，浮点除法和开方运算昂贵得要命。 然后，在《雷神之锤III》的源码里，大家发现了这么一段C语言代码： float Q_rsqrt( float number ) { long i; float x2, y; const float threehalfs = 1.5F; x2 = number * 0.5F; y = number; i = * ( long * ) &y; // evil floating point bit level hacking i = 0x5f3759df - ( i >> 1 ); // what the fuck? y = * ( float * ) &i; y = y * ( threehalfs - ( x2 * y * y ) ); // 1st iteration return y; } 注意那行注释：what the fuck? 这行代码的核心在于那个魔术数字：0x5f3759df。 写这段代码的人（有人说是卡马克，但考证后可能是其他大牛）利用了IEEE 754浮点数的二进制表示结构，通过一次整型位移和减法，直接猜出了一个非常接近的近似值，然后用牛顿迭代法稍微修了一下。 它的速度比系统的sqrt函数快了4倍。 当我第一次搞懂它的原理时，我意识到：真正的黑客，是在比特级别思考问题的。 他们看到的不是数字，而是内存里那一排排开关。 这种算法虽然现在随着硬件指令集升级已经不再需要了，但它代表的那种打破常规、利用底层特性榨干硬件性能的精神，是现在很多只会调包的程序员最缺少的。 5. 一致性哈希：分布式系统的定海神针 如果你做过后端，肯定知道负载均衡。 把请求分发给后端服务器，最简单的方法是 hash(key) % N，N是服务器数量。 这听起来很完美。直到有一天，你的一台服务器挂了，N变成了N-1。 或者你的业务爆发了，你需要加机器，N变成了N+1。 这时候灾难发生了：几乎所有的请求都会被重新映射。所有的缓存瞬间失效，流量像海啸一样打到数据库，你的系统瞬间雪崩。 一致性哈希用一个非常优雅的圆环结构解决了这个问题。 它把数据和服务器都映射到一个0到2^32的圆环上。数据顺时针找最近的服务器节点。 当一台服务器挂了，受影响的只有它是逆时针方向的那一小部分数据，其他数据不动。 当你加一台服务器，它只分担它逆时针方向那一部分压力，其他服务器不受影响。 我第一次接触这个算法是在看Memcached的客户端源码时。当时我就感叹，这简直是拓扑学在计算机工程里的完美应用。它把离散的哈希点连成了连续的环，用几何的思路解决了算术的问题。 现在所有的分布式存储、分布式数据库、包括像Dynamo、Cassandra这些顶级的系统，底层都在用这个思想。 如果你想深入了解分布式系统，我推荐去读一下Amazon Dynamo的经典论文 Dynamo: Amazon’s Highly Available Key-value Store。虽然是十几年前的论文，但它是一致性哈希在工业界大规模应用的鼻祖，读完你会对所谓的高可用有全新的认识。 6. 反向传播：让机器学会下山 最后，必须得说说现在AI的基石：反向传播。 虽然现在大家天天炼丹，PyTorch一个 .backward() 就完事了，但当你深入理解它背后的链式法则和梯度下降时，你还是会被感动。 想象一下，你被蒙住双眼放在一座高山上，你的目标是下到山谷底。你不知道方向，但你能感觉到脚下的坡度。 如果你往某个方向迈一步，感觉是上坡，那你就退回来；如果是下坡，你就大步走。 神经网络训练就是这个过程。 前向传播是机器在瞎猜。 损失函数告诉机器猜得有多离谱。 反向传播就是把这个“离谱程度”一层一层往回传，告诉每一个神经元：是你小子把结果搞坏了，你得调整一下你的参数（权重）。 这个算法的惊艳之处在于，它把学习这个极其抽象的概念，具象化成了求导数。人类的智能是基于经验的，而机器的智能是基于微积分的。 它简单到只需要高中数学就能推导，但强大到能支撑起现在的ChatGPT。 对于想入门AI的朋友，不要一上来就去啃西瓜书里的公式。我强烈推荐3Blue1Brown的深度学习系列视频。他在B站和YouTube都有号。他用动态的几何图形展示了梯度是如何在网络中流动的，看完你会有一种原来如此的通透感。 写了这么多，其实你会发现，这些惊艳的算法都有一个共同点： 它们往往不是在硬算，而是在“偷懒”。 HyperLogLog偷懒不存全量数据，改玩概率。 Bloom Filter偷懒不追求100%准确，改换速度。 Word2Vec偷懒不搞复杂语法规则，改算距离。 一致性哈希偷懒不搞全局重排，改搞局部调整。 作为一名在IT行业摸爬滚打多年的老兵，我最大的感触是：算法的本质，是对问题维度的重构。 刚入行时，我觉得算法就是写出更复杂的逻辑来覆盖所有情况。 后来我才明白，高阶的算法是跳出当前的逻辑维度，用概率、用几何、用物理的思维去降维打击。 现在的年轻人很焦虑，觉得算法岗卷，觉得模型都封装好了没啥可学的。 其实不然。工具变了，API变了，但解决问题的思维模式是不变的。 当你面对一个棘手的业务场景——比如如何在有限的内存里去重，如何在分布式的环境下保持一致，如何在海量数据里挖掘关系——你能脑子里蹦出这些经典的算法思想，并且知道怎么因地制宜地魔改它们，那才是你不可替代的核心竞争力。 这也是为什么我总是建议新人：多看论文，多读源码，少背面试题。面试题能让你进大厂，但对数据和算法的深刻洞察，才能让你在这个行业里走得远，走得稳。保持好奇心，这个世界可以用代码算出来，这事儿本身就挺惊艳的，不是吗？ 发布于 2026-01-29 19:55・四川 ​ 赞同 1822​ ​ 63 条评论 ​ 3350 ​ 95 ​ 分享 ​ 收起​ 更多回答 好地方bug​ 数学话题下的优秀答主 ​ 熟悉搜索的小伙伴已经快掏出自己的深度优先搜索跃跃欲试了。没错，此时搜索已经完全能解决丧彪赢还是输的问题。 我们不难得到一个非常暴力的搜索，只需要递归考察每个后继局面是必胜还是必败： 嗯，很好。我们拿来实验一下。 光是计算5，3，6这个简单的开局就用了接近2秒。 而计算5，3，6，2这个局面，就更困难了： 有小伙伴一定会意识到一个问题，我们在搜索的时候一定有很多重复的局面。例如对于5，3，6这个局面，我先搜索第一堆取2个，第二堆再取1个。和我先搜索第二堆取1个，第一堆再取2个。是一样的。 所以我们可以优化一下，例如通过哈希（这里我们用一种简单的质数哈希的办法），记忆每个局面，搜过了就不用再搜了。 嗯快了好多！！但是这就是我们想要的解决方案了吗？我们不妨分析一下复杂度： 每个石头都有可能减少任意数值，所以总的局面数是 \prod_{i=1}^na_i 。所以复杂度是 O(n\prod_{i=1}^na_i) 。当 a_i 相当大的时候，或者只要 n 稍稍一多，复杂度不可避免爆炸！ 想必绝顶聪明的丧彪和小美，脑袋里也不会跑这种，等到大道都磨灭了，都算不出来的算法。 按位异或 我们回顾一下上面的搜索，搜索本身是在刻画我们对先手胜败的论断，而我们对搜索的优化，本质上是在对局面进行编码计算。 那有没有办法把这两者结合一下呢？即，我们能找到一种对局面的计算，使得能够刻画我们先前提到的关于先手胜败的论断。 而这个计算就是：按位异或（bitwise-xor），我们用 \oplus 来表示。 什么是异或：异或是简单的0 1计算规则满足： 0 1 0 0 1 1 1 0 即相同则返回0，不同则返回1。 什么是按位异或：对于两个 m 位（不足 m 就补 0 ）二进制数， x 和 y 。我们定义 x 和 y 的按位异或为 x\oplus y=(x_{m-1} \text{xor } y_{m-1}, x_{m-2} \text{xor } y_{m-2},\cdots,x_0 \text{xor } y_0)_2 。也即每一位依次异或。例如 5=(101)_2, 3=(011)_2 ，则 5\oplus 3=(110)_2=6 。 按位异或有什么性质：注意到0 xor 0 = 0 且 1 xor 1 = 0，所以如果 x=y 则 x\oplus y=0 ；类似的， x\oplus 0=x 。 这里我们给出这个神奇的定理。 定理：对于 n 堆石堆的Nim Game，若每堆石头的数量为 a_1,a_2,\cdots,a_n ，则先手必胜当且仅当 a_1\oplus a_2\oplus\cdots\oplus a_n \ne 0 。 证明： 我们通过验证按位异或是否满足我们前面对先手胜败的论断，来证明上述定理。 Case 1: 若场面上没有石头，即， a_1=a_2=\cdots=a_n=0 ，则 a_1\oplus a_2\oplus\cdots\oplus a_n = 0 ，先手败。 Case 2: 若 a_1\oplus a_2\oplus\cdots\oplus a_n = 0 ，且 \exists a_i > 0 。我们修改任意一个 a_i ，使得其值降低。则接下来的按位异或和必不等于0，同时也是一个合法的Nim Game行动（这是由于我们是在降低值）。即后继局面中，不可能还存在使得按位异或和还等于0的情况，即后继局面都是必胜的。所以当前局面必败。 Case 3: 若 a=a_1\oplus a_2\oplus\cdots\oplus a_n \ne 0 ，则我们考虑 a 的二进制。其一定有一位是 1 。我们把 a 最高位的 1 给拿出来，这个 1 一定来自于某个 a_i （总不能凭空产生一个 1 吧）。接下来，我们把 a_i 中所有对应在 a 中是 1 的位置全部翻转（这句话有点长，注意是翻转 a_i ）。这会导致两个结果： a_i 变小。这是由于我们之前考虑的是 a 的最高位，所以之后翻转的位置一定都低于这个最高位，而这个最高位已经被翻转成 0 了，所以 a_i 一定是变小了。 整个按位异或和变成了 0 。这是显然的，因为我们通过翻转 a_i 把 a 所有是 1 的地方也同时翻转了。 这意味着，第一，这是一步合法的Nim Game操作；第二，后继局面里存在按位异或和为 0 的局面，即后继局面里面有必败的局面。 故当前局面能通过一步合法操作，使得后继局面必败。故当前局面先手必胜。 QED 注意：咱们这里的证明实际上是个归纳证明，即假设更小的局面总是满足定理的。 利用这个性质，我们再重新设计算法就非常简单了： 可以看到，同样计算[5, 3, 6, 2]，我们利用异或和，要快很多很多。这时我们的复杂度来到了 O(n) 。 无偏博弈 现在我们来考虑更一般的无偏博弈问题。 什么是无偏博弈 类似Nim Game，无偏博弈（Impartial Game）需满足如下条件： 两个游戏者轮流执行步骤； 当有玩家不能动的时候，即无可执行的步骤时，游戏结束，并选出胜者； 局面和操作必须是有限的； 两个游戏者除开先后手外，没有任何区别； 完全信息，所有游戏者都能看到整个局势； 无随机行动。所有行动都确定性地将目前局势转变到下一个局势。 例如我们常见的象棋围棋等，就不是无偏博弈，因为双方除开先后手区别，所持有的棋子也不一样。 另外像麻将，扑克等桥牌也不是，因为不是完全信息，每个游戏者不能看清整个局势。 那除开Nim Game，有什么其他游戏是无偏博弈的呢？这里给出一些例子。 Chomp：一块 n\times m Game中的处理，我们需要一个指标，来刻画当前局面的输赢。 我们定义函数 \text{mex}(S) 为集合 S 中最小的没有出现过的自然数，即： \text{mex}(S)=\min\{x\}, (x\in \mathbb{N}, x\notin S) 例如 \text{mex}(\{2, 3\})=0 ， \text{mex}(\{0, 1, 2,4\})=3 。 接下来，我们定义一个函数 SG 对于一个单局游戏（例如Nim Game中的一堆石头，或者Notakto中的一个井字棋）的状态 x ： 若其无后继局面，即无路可走的必输局面，则 SG(x)=0 ； 若其后继局面为 y_1, y_2,\cdots,y_k ，则 SG(x)=\text{mex}(\{SG(y_1), SG(y_2),\cdots,SG(y_k)\}) 。 Sprague–Grundy 定理：对于一个有 k 个单独游戏组合而成的局面 s ，若其每个游戏的状态为 s_1, s_2,\cdots, s_k ，则局面 s 的 SG 值为： SG(s)=SG(s_1)\oplus SG(s_2)\oplus\cdots SG(s_k) 。且局面 s 先手必胜当且仅当 SG(s)\ne 0 。 有点绕是不是？我们举些例子： 比较容易观察到，对于Nim Game而言，每一堆石头的 SG
