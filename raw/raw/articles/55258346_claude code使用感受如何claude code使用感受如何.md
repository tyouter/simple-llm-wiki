CLAUDE.md 越短越好 立即可用 我见过一个 800 行的 CLAUDE.md，列出了每个文件夹的用途、每个 API 的参数、每个边界情况的处理。作者确实花了很多心思，但效果适得其反。 Claude 的注意力是有限的。上下文靠前的部分权重更高。我的经验法则是：单个 CLAUDE.md 控制在 200 行以内。官方文档说 CLAUDE.md 会完整加载，但更短的文件意味着更好的遵循度。太长会消耗上下文、降低合规性。溢出的内容应该放到 .claude/rules/ 或用 @imports。 CLAUDE.md 越长，Claude 越不听话。 我用 200 行作为个人上限，不是文档规定的硬限制。 该放什么：Claude 从代码里读不出来的「为什么」。架构决策、历史教训、业务约束、风格偏好。 不该放什么：API 文档、逐文件描述。Claude 自己会读代码。 我的全局 CLAUDE.md 只有 4 行： - commit 信息使用 conventional commits 规范 - 修改前先解释计划，等我的确认 - 优先简单方案，不要过度工程 - 不要重构我没提到的文件 就这 4 行，帮我省了上百次「不是，我不是这个意思」。 在项目层面，我放项目事实： # Ship Faster ## 架构 - 子代理的核心价值是上下文隔离。每个子代理有自己的上下文窗口。输出留在子代理里，只把摘要返回给你。Claude Code 支持自定义子代理，可配置工具权限和角色描述。Claude 会自动把任务委派给匹配描述的子代理，每个都在自己的隔离上下文中运行。 我用三个代理模板： explorer（只读，快速代码库搜索） planner（只读，研究和分析） executor（全能力，复杂任务） 自定义代理放在 .claude/agents/： --- name: code-reviewer description: 代码审查代理，只读 tools: Read, Grep, Glob, Bash model: sonnet --- 你是一个代码审查员。检查：安全漏洞、性能问题、可维护性。 只报告问题。不要修改代码，不要建议重构。 注意：子代理默认继承父会话的工具。想要真正只读的代理，必须显式设置工具白名单。只在描述里写「read-only」不会限制工具访问。 我用子代理最反直觉的方式：不是让它写代码，而是模拟用户。 我设置了 6 个 AI 人格扮演不同类型的「找茬者」，每个作为子代理任务委派出去。15 分钟后，他们找到了 5 个我自己用了两周都没发现的问题。 我做了一个 AI 配置系统，用了两周，感觉挺稳。但我不确定别人会怎么体验它。于是我创建了 6 个子代理，每个扮演完全不同的用户角色： 角色 关注点 怀疑型资深工程师 10 年经验，默认不信任新工具。关注安全、副作用、隐藏依赖 安全审计员 安全审计背景。关注密钥泄露、权限边界、远程执行风险 新接手维护者 刚接手项目，零背景知识。关注文档清晰度、上手路径 重度 CLI 用户 一个功能的描述和实际行为不匹配。重度 CLI 用户立刻失去信任 status 命令不是真正的只读。安全审计员第一个发现 通知管道没有存活探针。运维拒绝把它当作健康检查 修完每个问题后，同样的 6 个代理又跑了一轮。所有高优先级信任断点清零。剩余问题降到措辞和可观测性微调。 这种测试手工做要一周。AI 两轮完整测试只用了 15 分钟。 这不仅是测试技术。这是质量保证的新思路：让 AI 扮演你的用户，在发布前找到问题。 3. 好指令要精准 有一次我给了这个指令：「帮我优化这个函数。」 Claude Code 把函数拆成三个文件，加了一个工厂类，改了导入路径。我花了二十分钟 review，才发现没有一个是我想做的。 问题出在指令太模糊。好指令只需要三个要素：目标、约束、上下文。 目标: 把 getUserData() 响应时间从 800ms 降到 200ms 以内 约束: 不改接口签名，不加新依赖，只动这一个函数 上下文: @src/services/user.ts#L45-L89 验证: user.test.ts 必须全部通过 禁止: 不改数据库 schema，不加缓存层 @ 引用加行号让 Claude 精准聚焦。UI 问题，我直接 Ctrl+V 粘贴截图，比用文字描述布局准确十倍。 「不要」比「要」更重要。 「不要加新依赖」比列出 10 个允许的库更有效。Claude 有时会过于热情。你让它修个 bug，它把周围代码也重构了。说一句「只改这个函数」省很多麻烦。 探索性任务反而适合模糊指令。 「看看这个模块有没有潜在问题」比「检查这 5 个具体方面」能发现更多东西。实施时切回精准模式。 4. Plan Mode 是最被低估的功能 立即可用 有一次我让 Claude Code 直接重构一个 600 行的路由文件。它走到一半，引入了循环依赖，又试图修复循环依赖，结果越搞越糟。我按了 Ctrl+C，丢了半小时的上下文。 从那以后，复杂任务我永远用 Plan Mode。 三种进入方式： # 启动时 claude --permission-mode plan # 会话中（按 Shift+Tab 直到出现 Plan Mode） Shift+Tab # 重复按直到出现 # 自然语言 "先别改代码，给我一个实施计划" Plan Mode 下，Claude 只输出计划不执行任何操作。关键操作：按 Ctrl+G 在编辑器里打开并编辑计划文本。删掉不必要的步骤，加约束，确认后 Claude 按更新后的计划执行。 改计划一句话的事。改已经写好的代码要花十倍时间。 标准四步 workflow： Explore（Claude 读代码）→ Plan（输出计划）→ Implement（逐步执行）→ Verify（跑测试） 不需要 Plan Mode 的情况：只有一种可能实现的任务。重命名变量、修错字、加一行日志。不用绕弯。Plan Mode 是有多个选择的任务。 5. 上下文管理是真正的瓶颈 本周设置 有一次我任务进行到一半，跑了 /context 看到 78% 使用率。然后 Claude 突然开始重复我们之前否掉的方案，像失忆了一样。从那以后我把 /compact 变成常规习惯。 Claude Code 支持非常大的上下文窗口，但塞满的上下文意味着质量急剧下降。 塞满的上下文不是超限失败，是注意力分散失败。 五个管理策略： /clear：全部清空，完全切换任务时用 **/compact [focus]**：压缩历史但保留关键点。继续同一任务时用 /context：查看当前上下文使用率。知道自己在哪 子代理：单独模块用单独代理。别污染主会话 压缩保留****指令：在 CLAUDE.md 里告诉 Claude 压缩时保留什么 Claude Code 这里其实有两个机制：CLAUDE.md 文件和 auto memory。在 CLAUDE.md 里，我把项目指令当作共享项目事实和团队标准，用户指令（~/.claude/CLAUDE.md）当作跨项目跟随我的个人偏好。Auto memory 在 ~/.claude/projects/memory/ 里，会话开始时加载 MEMORY.md 的前 200 行。输入 /memory 查看加载的 CLAUDE.md 文件，管理 auto memory，编辑 memory 条目。 6. 收紧你的权限 有一天，Claude 修 bug 的时候顺手删了一个「看起来没用」的配置文件。那个文件是生产环境的密钥引用条目。 那之后我花了半小时配置 .claude/settings.json： { "permissions": { "deny": [ "Bash(rm -rf*)", // 禁止：递归强制删除 "Bash(*--force*)", // 禁止：任何带 --force 的命令 "Edit(.env*)", // 禁止：编辑环境变量文件 "Edit(*.pem)" // 禁止：编辑证书文件 ], "ask": [ "Bash(git push*)", // 询问：推送到远程仓库 "Bash(npm publish*)", // 询问：发布 npm 包 "Edit(src/config/*)" // 询问：编辑配置目录 ], "allow": [ "Bash(npm run*)", // 允许：运行 npm 脚本 "Bash(git commit*)", // 允许：提交代码 "Bash(git add*)", // 允许：暂存文件 "Read(**)" // 允许：读取任何文件 ] } } 优先级：deny > ask > allow。Deny 最高。如果命令匹配 deny，就算也匹配 allow 也不会执行。 Settings 文件有自己的层级：托管（组织策略）、CLI 参数、本地（.claude/settings.local.json，不提交）、项目（.claude/settings.json，共享）、用户（~/.claude/settings.json）、默认。一个重要细节：权限列表这类数组设置会跨范围合并而不是替换。所以用户层的 deny 规则和项目层的另一个 deny 规则都会生效。 我在项目层 deny 业务风险，比如删除迁移。我在全局层 deny 安全基线，比如执行 rm -rf。两层，独立维护。 只读****就是只读。 没有「顺便」。我的第一个 status 命令版本偷偷调了 sync，理由是「检查时顺便同步一下」。跑了 6 人格测试后，安全审计员立刻发现：「你的只读脚本有副作用。我没法信任它的输出。」副作用摧毁可信度。 7. Hooks：把规则变成法律 本周设置 我在 CLAUDE.md 里写了「提交前跑 lint」。Claude 有时会跳过。文件里的规则是建议。AI 有时会觉得「这个情况特殊」就忽略了。 CLAUDE.md 里的规则是建议。Hooks 是代码级强制，永远不会忘。 四种 hook 类型： 类型 说明 command 执行 shell 命令，从 stdin 读 JSON http 发 POST 请求到 URL prompt 做 LLM 调用判断 yes/no agent 生成子代理验证 三个实用场景： 自动格式化：每次编辑后跑 Prettier { "hooks": { "PostToolUse": [{ "matcher": "Edit|Write", // 匹配编辑或写入操作 "hooks": [{"type": "command", "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"}] // 自动格式化 }] } } 保护关键文件：阻止修改生产配置 { "hooks": { "PreToolUse": [{ "matcher": "Edit|Write", "hooks": [{"type": "command", "command": ".claude/hooks/protect-files.sh"}] }] } } 退出码 0 表示允许，退出码 2 表示阻止。但只有带决策控制的事件才能真正阻止动作。PreToolUse 可以在工具调用前阻止。PostToolUse 不行，因为工具已经执行了。 压缩后重新注入：长对话被压缩时，重要指令可能丢失。Hook 自动重新注入： { "hooks": { "SessionStart": [{ "matcher": "compact", "hooks": [{"type": "command", "command": "echo 'Use Bun, not npm. Run bun test before committing.'"}] } } Hooks 的真正价值是把工程标准变成系统约束。自动格式化只是附赠。人会忘规则。代码不会。 8. 让 AI 从你的行为中学习（可选） 这是高级实验，非必需。如果只是想用好 hooks，上一章就够了。 反向思路：让 AI 从我的行为里提取规则。流程： AI 写初稿 → 我改到满意 → observe.py 记录原版/终版 → improve.py 对比两版 → 提取「我改了什么」 → 规则按置信度排名 P0/P1/P2 → 人工审核后，P0 规则写回 SKILL.md # observe.py：记录 AI 初稿和我的终版 python3 observe.py record-original draft.md # ...我做修改... python3 observe.py record-final final.md # improve.py：从 diff 提取规则 python3 improve.py auto --skill . # 自动对比，提取变更模式，生成规则候选 # P0（出现 5 次以上）推送人工确认后写入 # P1 低频，归档 # P2 仅记录 结果：48 条规则，740 行。循环越多，AI 写得越像我。 我还设置了 cron job 每晚自动跑提取，不用手动触发： 0 23 * * * python3 ~/.ai-core/skills/scripts/improve.py auto --skill . 限制：自动学习不是魔法 核心安全规则，比如权限边界、密钥保护、破坏性命令阻止，必须手写。不能委托给自动化。 偶然的修改会被误提取成规则。有一次我临时改了文件缩进因为那个文件风格不一致。improve.py 提取成「总是用 2 空格缩进」。那条规则是错的，因为项目标准是 4 空格。 所有 P0 规则必须经过人工审核才能写入。确认它们反映真实偏好，不是一次性修改。自动学习只处理风格偏好。核心安全规则必须手写。 9. Git Worktree：零风险实验 我以前重构前会犹豫。万一搞砸了怎么办？Git 分支共享工作目录。切分支会影响当前状态。 Worktree 不一样。每个分支有自己的独立工作目录： # 创建实验 worktree claude --worktree experiment/refactor-auth # 或者手动 git worktree add ../project-experiment feature/new-auth Claude 在实验目录里想干什么干什么。主目录原封不动。如果不成功，git worktree remove，零成本。 创建一个 worktree 只要几秒。失败的实验要几小时。 Git 规则： 不要让 Claude 自动 ****push。它可以 commit，但 push 需要我确认。一旦 push 到远程，回滚成本就高了 频繁 commit。每个原子步骤后都 commit。/rewind 可以回滚到任意检查点 注意破坏性操作。git reset --hard、git push --force、rm -rf。批准前三思。这些操作没有撤销 从 PR 恢复上下文。claude --from-pr 123 自动加载 PR 的变更和讨论 所有探索性任务、架构重构、不确定的大改动都走 worktree。 10. Extended Thinking：按需调整 立即可用 思考默认开启。Claude Code 总会做一些推理。但努力程度不应该总是拉满。简单任务高努力纯粹浪费 token，而且推理过程消耗上下文，影响后续对话。 有一次我设计一个复杂的权限系统，涉及多角色、继承、动态规则。默认努力产生的计划有明显漏洞。切换到 /effort high 后，它思考了近一分钟，拿回一个完全不同的架构，覆盖了我没想到的边界情况。 # 调整推理深度 /effort high # 更深推理 /effort low # 简单任务，省 token 我的规则：只有改动超过 3 个文件时才提高努力。单文件编辑，默认就好。架构设计、复杂 bug 诊断、需要多约束推理的任务，值得花 token。 11. MCP 不要过载 MCP（Model Context Protocol）让 Claude Code 连接外部服务，比如数据库、支付平台、GitHub。安装一个 MCP server，Claude 就能在对话里直接操作那个服务。 每个 MCP server 都会给你的上下文加工具定义，开销会累积。我的经验，每个 server 大概消耗 100 到 500+ token，具体数值因配置而异。我试过同时跑 8 个。有效上下文明显下降，代码生成质量变差。 每多一个 MCP server，有效上下文就少一点。按项目添加，用完删除。 我只保留 2 到 3 个： claude mcp add supabase -- npx @supabase/mcp-server-supabase --project-ref $REF claude mcp add stripe -- node /tools/stripe-mcp/index.js .mcp.json 配置： { "mcpServers": { "supabase": { "command": "npx", "args": ["@supabase/mcp-server-supabase", "--project-ref", "xxxxx"] }, "stripe": { "command": "node", "args": ["/tools/stripe-mcp/index.js"] } } } 连上 Supabase 和 Stripe 后，我在对话里说「查一下最近 7 天失败的支付」。Claude 直接查 Stripe，返回结果，分析。以前要写一次性脚本。现在就是一句话。 用 /mcp 管理服务器连接，禁用不需要的。 12. 不要盲目信任 Claude 的代码 三个常见问题： 过度工程：让它写个简单工具函数，它建一个完整的抽象层，泛型、接口、工厂模式全上。解决：在 CLAUDE.md 里写「优先简单方案」 幻觉 ****API：用不存在的 SDK 方法。名字看起来合理，文档说不是。解决：跑测试 范围蔓延：改一个函数，它动五个文件。解决：明确说「只改 X，别的别动」 写作者/审查者双会话模式：一个会话写，一个会话审。角色分离，互不干扰。 一个会话写代码，另一个审查。审查者不知道代码怎么生成的。它只看结果提问。两个会话，角色分离，无交叉污染。 # 审查者会话指令 你是一个代码审查员。这段代码有没有： 1. 幻觉 API（不存在的方法） 2. 竞态条件 3. 任务范围之外的修改 只报告问题。不要建议修复。 @src/auth/service.ts 审查者发现的问题返回给写作者修复。通常 2 到 3 轮，质量比单会话好很多。 13. Claude Code 不是万能的 有一次我让 Claude Code 配置一个我一无所知的 Kubernetes 集群。它跑起来了。能用了。我一个字都看不懂。两周后出问题，我完全不知道从哪下手。这是我犯过的最贵的错误之一。 Claude 能生成任何代码，但如果你看不懂，你就接不了手。 四条红线： 不懂的事别做。Claude 能生成任何东西，但如果你读不懂，你就维护不了。生产环境里不懂的代码既是技术债也是安全风险 没有版本控制别用。没有 Git 就没有回滚。Claude 的修改可能覆盖你的文件。没有版本控制，就是盲飞 别扔大任务。「重构整个认证系统」是灾难级提示。「从 validateToken 里把 JWT 解析抽成独立函数」才是正确姿势。小任务更准确，范围可控 别指望一次完美。初稿永远是初稿。/rewind 回滚，给精准反馈说哪里不对。迭代才是正确方式 这四条红线不是限制 AI，是保护你的判断力。你不懂就交出去的每个决定，都在消耗你对系统的控制权。 14. 进阶：多系统协作 如果你只用一个 AI 工具，不跑自动化服务器任务，这章对你几乎没价值。直接跳到总结。这章讲的是我把 Claude Code 放进更大 workflow 之后做的事。不是通用实践。 我遇到的问题很具体：Claude Code 和
