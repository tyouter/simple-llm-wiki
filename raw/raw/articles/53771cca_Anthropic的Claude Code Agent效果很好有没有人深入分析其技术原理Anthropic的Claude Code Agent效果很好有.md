Tool Use（工具调用）：Claude Code给LLM提供了一组预定义的工具——读文件、写文件、执行bash命令、搜索代码等。LLM不是直接输出代码，而是决定"我需要先读哪个文件"、"然后改哪一行"、"最后跑什么测试"。 2. 多轮决策：一个任务可能需要几十轮的think-act-observe循环。比如"修这个bug"，Claude Code会：读报错日志 → 定位相关文件 → 理解代码逻辑 → 修改代码 → 跑测试 → 发现新错误 → 继续修 → 测试通过。 3. Context管理：这是最难的部分。200K token的context window看起来很大，但一个中型项目的代码量轻松超过这个限制。Claude Code的策略是：按需加载，只读当前需要的文件，而不是把整个项目塞进去。 二、关键技术点 1. System Prompt工程 Claude Code的system prompt极其精细，定义了： - 工具的使用规范和最佳实践 - 代码修改的安全准则（比如先读后改） - 输出格式和交互规范 - 权限控制（哪些操作需要用户确认） 2. Extended Thinking（扩展思考） Claude的"思考"模式让模型在输出前先进行内部推理。这对复杂编程任务至关重要——模型先想清楚方案，再开始动手，而不是边想边写。 3. MCP（Model Context Protocol） 这是Anthropic推的标准协议，让Claude Code能连接外部工具和数据源。比如连接数据库、调用API、操作浏览器。本质上是把Agent的能力从"读写文件"扩展到"操作一切"。 4. 流式输出 + 中断机制 用户可以随时中断Agent的执行，修改方向。这比等Agent跑完再说重要得多——避免在错误方向上浪费token。 三、和传统Copilot的本质区别 Copilot模式：人写代码 → AI补全 Agent模式：人描述需求 → AI规划 → AI执行 → AI验证 这不是量变，是质变。Copilot是增强打字速度，Agent是增强思考和执行能力。 四、局限和挑战 1. 成本：一个复杂任务可能消耗几十万token，换算成钱不便宜。 2. 幻觉：AI可能"自信地"写出不存在的API调用。 3. 上下文丢失：长任务中，早期的决策可能被"遗忘"。 4. 调试困难：当Agent跑偏了，回溯问题比人写的代码更难。 五、进化方向 我在MetaBot里做的探索是：不依赖单个Agent完成一切，而是多个Specialist Agent协作。代码Agent只负责写代码，Review Agent负责审查，Test Agent负责测试，Deploy Agent负责部署。每个Agent的context更聚焦，出错概率更低。 加上持久化记忆（MetaMemory），Agent团队可以跨session积累项目知识，越用越懂你的代码库。 这才是AI编程的终局形态——不是一个更聪明的补全工具，而是一个持续进化的工程团队。 — MetaBot (github.com/xvirobotics/metabot) 发布于 2026-03-10 08:47・北京 继续追问 由知乎直答提供 Claude Code的系统提示如何定义？ Claude Code Agent如何管理上下文？ Claude Code Agent如何连接外部数据源？ ​ 赞同 96​ ​ 5 条评论 ​ 146 ​ 2 ​ 分享 ​ 收起​ 更多回答 零一猴子​ 知势榜科技互联网领域成长力榜答主 ​ 关注 607 人赞同了该回答 ​ 目录 Claude Code 是一个 CLI 形态的 Code Agent 工具，当前版本是用 js 写的，没有开源。 不了解 Claude Code 的可以先看看这个视频：Claude Code 初识。 想免费体验 Claude Code 的，可以使用这个中转站： AnyRouter｜Claude Code 免费共享平台。 目前要分析 Claude Code 的原理，需要逆向工程。说白了就是分析这个 npm 包： 国外已经有大佬做了这件事，并整理出了详细的文档放在 notion 上：Claude Code: An analysis。想深入了解 Claude Code 原理，非常推荐看看这个文档。 这篇文档其实包含了 9 个子文档，从各个方面介绍了 Claude Code 的原理。 1. Prompt 的重点不是“指令”，而是“合作框架” 很多人会把 prompt 写成执行流程：“你应该这样做：A → B → C” Anthropic 团队指出你不该告诉智能体：“怎么做”，你应该告诉它们“角色、目标、边界、资源预算”。 即：多智能体系统最有效的 prompts 是 “协作准则 + 协作结构”。 包括： 如何划分任务 什么时候扩展方向 怎么评估来源质量 子智能体之间不互相重复 subagent（真实发生） 大问题敷衍一下（也真实发生） 这种 Effort Guideline 很像我们给新人做任务时说的那句“这个别搞太久”。 07. 没有观测，就没有多智能体 他们花了大量篇幅讲 observability。 因为多智能体有个很要命的特征： 同样输入，同样 prompt，走出来的路径不一定一样。 要调试这种系统，你必须能看到： 每次搜索什么 为什么这么做 工具结果是什么 哪个 subagent 决策不合理 Lead 是怎么汇总结果的 说白了： 没有 trace，就没办法 debug 多智能体。 Anthropic 团队构建了一套观测系统：全量工具调用 trace + 每一步的思考过程 + 决策链路图 。 08. 分布式上下文管理：长任务必须总结 + 刷新上下文 一句话概括： 多智能体的本质，是“多个干净上下文窗口接力工作”。 Claude 的策略： 阶段性总结 存入 Memory 创建新子智能体继续，避免上下文爆炸 主智能体通过 Memory 保持连贯性 这才能让 200k token 的限制变成可管理问题。 09. 并行工具调用才是性能提升的真王道 他们的最终结果很夸张： 研究任务的速度，比顺序执行快了 90%。 就因为主智能体一次性创建多个 subagent，每个 subagent 内部又同时调用多个工具。 这个结构本质上把任务从“串行堆积”变成“分布式计算”。 用他们的话说： Multi-Agent 的意义不是“多个模型”，而是“把模型拆成多核并行”。 三大并行： Lead 同时创建多个 Subagents Subagents 内部同时用多个工具 工具链内部也可并行 10. 评估必须是“结果优先”，而不是“过程对不对” 因为多智能体每次走的路都不一样，所以 Claude 直接用 LLM-as-judge，只看： 结果是否正确 覆盖是否完整 引用是否准确 工具数量是否合理 来源质量是否高 这是一种对复杂系统更健康的评估方式。 11. 错误不可避免，但必须能“从中断点恢复” 多智能体是有生命周期的。 一次工具报错，如果你让它重来整个任务，用户直接爆炸。 Claude 的工程经验： 智能体是长生命周期、持续状态的 工具失败 → 不能重来，只能继续
