# Vibe Coding

## Definition
An approach to AI-assisted programming where high-level aesthetic, emotional, or stylistic descriptions ("vibes") are used as the primary prompt to guide the generation of code, particularly for frontend interfaces and visual design.

## How It Works
Instead of providing detailed, technical specifications ("create a div with flexbox"), the programmer gives the AI a descriptive, often subjective direction. Examples include:
- "Make a landing page that feels like a cozy, independent bookstore in the rain."
- "Generate a UI with a cyberpunk, neon-drenched aesthetic."
- "Code a dashboard that has the clean, airy vibe of a Scandinavian design magazine."

The AI (like [[Claude Code]]) then interprets this "vibe" and produces corresponding HTML, CSS, and JavaScript to realize it.

## Connection to Curated Design Prompts
Platforms like [[designprompts.dev]] operationalize [[Vibe Coding]] by providing concrete, pre-engineered XML prompts that encapsulate specific "vibes" (e.g., "Brutalist," "Retro Futurism"). Users can select a vibe from a gallery and get a prompt that reliably generates code in that style, making the technique more accessible and consistent.

## Role in AI-Free Aesthetics
[[Vibe Coding]] is a key technique for achieving [[AI-Free Interface Aesthetics]] because it steers the AI using creative, human-centric direction rather than letting it fall back on its most common, generic training outputs. It prioritizes feel and atmosphere over mechanistic instruction.

**See Also:** The workflow in [[Using DesignPrompts.dev to Generate High-Quality AI-Free Interfaces with Claude Code]] is a formalized example of Vibe Coding using a curated prompt repository.

## 在Claude Code指南中的批判
在[[Claude Code高效配置与开发实践指南]]中，[[yahah]]等答主对Vibe Coding提出了明确批判：

### 核心问题
1. **缺乏中间设计环节**：将自然语言直接"投影"为编程语言，跳过必要的规格说明和规划阶段
2. **代码质量低下**：容易导致结构混乱、边界条件缺失、与整体架构脱节
3. **不可靠性**：依赖即时感觉而非系统化思考，产出不稳定

### 对立方法论
[[yahah]]提倡用[[Spec Coding]]替代Vibe Coding：
- **Spec Coding**：先有明确规格说明，再有详细[[Plan]]，最后生成代码
- **强调工程性**：提高确定性、可验证性和架构合理性
- **反对感觉驱动**：用系统方法取代模糊感觉

### 与Agent-First Development的关系
[[Agent-First Development]]范式也与Vibe Coding对立。智能体优先的开发要求清晰、结构化的输入，而非模糊的"感觉"描述。Vibe Coding本质上是人类中心、感觉驱动的，而Agent-First是智能体中心、规范驱动的。

## 实践中的表现
典型Vibe Coding示例：向AI发出"给我写个登录功能"这样模糊的指令，然后根据AI的即兴输出不断修补。这种方法被批评为低效和不可靠，在严肃工程项目中应避免使用。

## Additional Insights



在[[CLAUDE.md高效编写指南与AI编程实践]]的讨论中，[[yahah]]等贡献者强烈反对纯粹的Vibe Coding，认为它缺乏结构性和可验证性。与强调预先规划和详细规格的[[Spec Coding]]相比，Vibe Coding更依赖即时交互和探索，这在复杂项目中可能导致代码质量问题和架构不一致。

[[CLAUDE.md配置哲学]]虽然不直接禁止Vibe Coding，但其“少即是多”、强调确定性的原则更倾向于支持结构化的方法。Vibe Coding可以作为[[Agentic Loop]]的一种实现形式，在创意探索阶段有用，但在生产代码开发中应谨慎使用。

文档建议避免纯粹的Vibe Coding，而是采用更结构化的[[Spec Coding]]方法，特别是在团队协作和长期维护的项目中。Vibe Coding更适合原型设计、创意探索等非关键任务，而Spec Coding更适合需要高质量、可维护代码的生产环境。
